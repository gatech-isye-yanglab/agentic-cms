# seed/ — The Prompt-First Compression of agentic-cms

This artifact ships in two forms.

The **full repo** (the parent `agentic-cms`) demonstrates the
methodology end-to-end and is what reviewers verify directly: builds
the synthetic CMS database, runs the gold-standard pipelines, exercises
the multi-agent prototype, ships the saved traces.

This **seed** (~hundreds of KB, vs ~1 MB of generated code) is the
prompt-first compression. An agent at any institution can clone the
seed, run its own LLM, and decompress it into a schema-adapted
equivalent of the full repo. We measure decompression fidelity
behaviorally, not by byte-equality: the regenerated code passes the
synthetic-DB compliance suite, and produces the same row counts at
each pipeline stage within tolerance.

## Why a seed at all

If the full repo's code was generated by an agent from a set of
prompts, then the prompts are the actual information; the code is
their decompression. Three categories of content live in the full
repo:

| Category | Compressible? | Where it lives in the seed |
|---|---|---|
| **Method** — agent-regenerable (SQL pipelines, agent loop, skill files, all docs) | Yes, ~95% | [`prompts/`](prompts/) |
| **Data** — empirical inputs (institutional schema export, validated code lists) | No — must be carried | [`data/`](data/) |
| **Evidence** — recordings (saved traces, row counts, deterministic hashes) | No — must be carried | [`evidence/`](evidence/) |

A "minimum information" repo isn't *prompts only*. It's
*prompts + data + evidence*.

## How an agent uses this

```bash
# Inside an empty directory:
git clone <seed-repo> .

# Start your favourite coding agent — Claude Code, Codex, Cursor, etc.
claude   # or `codex`, `cursor .`, ...

# The agent reads AGENTS.md, then prompts/01..08 in order.
# At each step it generates the corresponding artifact in this directory.

# After the agent finishes:
bash verify.sh
```

If `verify.sh` exits 0, the seed has decompressed successfully into a
behaviorally-equivalent copy of the full repo.

## How a human uses this

Read [`prompts/`](prompts/) in order. They're written to be readable as
engineering specifications, not as code dumps. They describe **what**
each artifact does, **why** it exists, **what's tricky**, and **how
to verify** it. The actual full-repo code is also a valid lens —
each prompt has a "see also" pointer to the existing files it
regenerates.

## Honest limits

- **Non-determinism.** Two LLM runs of the same prompt produce different
  code. Verification is behavioral (does the regenerated suite pass?),
  not byte-identity.
- **Model drift.** As LLM models change, the same prompts produce
  different code, possibly worse. We pin the original-run model in
  [`evidence/traces_v1.txt`](evidence/traces_v1.txt) for reference.
- **Compounding error.** Later prompts depend on earlier outputs.
  An error in prompt 01 can cascade. Empirically open whether this
  stabilises or diverges over 8 prompts.
- **Data is irreducible.** [`data/columns_formats.csv`](data/columns_formats.csv)
  is your institution's schema export — you cannot infer this from a
  prompt. It must travel with the seed.

## Status

This is a **research experiment** alongside the canonical full-repo
artifact. The compression ratio achieved here, and the
decompression-fidelity score on first agent regeneration, are
themselves a contribution worth measuring. See the parent
[`README.md`](../README.md) and [`docs/architecture.md`](../docs/architecture.md)
for the project as a whole.
